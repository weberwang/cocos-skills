# Cocos Project Planning

Convert approved inputs into the sole `.cocos-workflow/artifacts/implementation-plan.md` consumed by production and integration. Plan only: do not generate assets, write game code, call Cocos MCP, or modify `workflow.yaml`.

## Validate inputs

1. Read the orchestrator `workflow-contracts.md`, `state-machine.md`, `mcp-safety-policy.md`, [the plan contract](references/implementation-plan-contract.md), and [the scene-boundary contract](references/scene-boundary-contract.md).
2. Read `.cocos-workflow/project-profile.yaml`, `requirements.md`, approved systems and technical design artifacts, and the frozen visual-direction artifact. Require approved project configuration, requirements, systems design, technical design, and visual direction; formal scene concepts are planned here and produced later inside `production`.
3. Compare the current requirements, systems-design, technical-design hashes, and `visual_direction.version`/`content_hash` to their approval records. On any missing or mismatched input, return `blocked` or `stale`; do not create an executable plan.
4. Accept only orchestrator-authorized plan, result, and report paths. The orchestrator is the only writer of `workflow.yaml`.
5. When the task declares a decision-changing update to an approved plan, require a `$grilling` confirmation whose stage and subject hash match `decision_change`; on a missing or mismatched confirmation, return `blocked` and do not create a replacement plan.

## Create the plan

Production order is fixed: **core gameplay prototype → human confirmation → module decomposition → global scaffold → business-flow levels with formal scene loops**. Do not invert this order.

1. Map every requirement page to a scene or UI state with entry, exit, core interaction, acceptance criteria, and its planned Pencil-draft and high-fidelity-concept task IDs (for the later formal loops). 将技术设计的 `scene_structure` 落为每个场景唯一的 `scene_blueprint`：完整节点树、稳定节点 ID、父子关系、必需/条件组件、关键属性、对应模块和读回断言。蓝图必须包含场景根、运行时根、Canvas/UI 根、游戏内容根；UI、安全区、HUD、弹层、相机和输入节点按已声明条件纳入，不能只列 `node_ids`。
2. Define exactly one `vertical_slice` as the **first** production gate: select the smallest set of core gameplay `scene_ids` that lets a player complete `start → challenge → resolution`. Set `implementation_mode: prototype`. Plan lightweight `core-gameplay-code → prototype integration → human verification → vertical-slice-review` tasks (placeholder/minimal assets allowed). The prototype integration must declare `integration_mode: prototype`. Do **not** require Pencil drafts, high-fidelity concepts, module decomposition, global scaffold, or a formal asset manifest for this prototype. Map each prototype scene to a later formal `scene_loop` id in `formal_scene_loop_ids`; do not put execution status or approval into the plan.
3. Only after the vertical-slice review task, create the 模块拆分任务 (`kind: module_decomposition`). Define stable module IDs, `business_flow_level`, responsibilities, public interfaces, owned paths, dependency direction, test boundaries, and acceptance criteria. The task must depend on `vertical-slice-review` and output approved `module_decomposition` and `dependency_graph`; do not turn unconfirmed gameplay into formal module detail.
4. Create one global scaffold code task (`kind: global_scaffold`) after approved `module_decomposition` and before every formal scene loop. It owns only shared bootstrap paths and establishes app entry, scene routing, global state/events, configuration, resource service, error boundary, and test seams. Its acceptance checks must pass before any formal scene-specific code, integration, or human verification starts.
5. Define `business_flow_levels` from the user-visible business flow for **formal** implementation only. Start at level 1 and use consecutive levels only; every level lists its module IDs, page IDs, and completion task IDs. Modules, pages, and tasks in a higher level must wait for every completion task in the preceding level; tasks in the same level may run in parallel only when paths do not overlap. Prototype tasks are excluded from level gates but must finish before any level-1 formal task.
6. Build an asset dependency graph including source, import target, consumers, target dimensions, license status, and missing-asset handling. Each formal scene owns one pending `artifacts/game-assets/<scene_id>.yaml`; Cocos import and every runtime binding must wait for the exact approved per-scene manifest hash.
7. Generate `.cocos-workflow/artifacts/capture-manifest.yaml` from all frozen `project-profile.capture_profiles`. It is a human-review script: bind the frozen initial scene and define the human reviewer, manual editor-preview steps, required screenshots/logs, expected observations, approved baseline paths, optional masks, and pixel-diff thresholds for `mobile-small`, `mobile-standard`, and `mobile-large`. The workflow must not launch or operate any runtime; no profile may be omitted.
8. Split formal production into non-overlapping asset-preparation and code tasks. Every formal `kind: code` task depends on the approved `module_decomposition` task and names only assigned `module_ids`. Scene-specific formal code also depends on the passed `global_scaffold` task and the vertical-slice gate. Formal code files may run in parallel only when they belong to the same business-flow level and `allowed_paths` do not overlap; a code binding task that names `asset_ids` depends on the approved asset-manifest gate; asset preparation must not use the Cocos editor.
9. Define formal 场景小循环 (`scene_loops`): each scene loop owns one `scene_id` and `business_flow_level`, sets `is_core_gameplay` when the scene is in `vertical_slice.scene_ids`, depends on `vertical-slice-review` + `module_decomposition` + `global_scaffold_task_id`, and orders `scene-boundary-grilling → pencil-draft → visual-concept → asset-preparation / code → human-review` inside `production`. The boundary task must produce the human-approved `artifacts/scene-boundaries/<scene_id>.md` from the scene blueprint and frozen upstream inputs before any scene design; Pencil, high-fidelity concept, asset-preparation and code must carry its hash. Each `visual-concept` task owns exactly one scene and one final image; order scene loops serially by making every later boundary task depend on the prior scene's human review so effect images are never batch-generated. When a loop is core gameplay, it is the **formal version** that replaces the prototype—never ship the prototype as the deliverable. The Pencil draft must depend on the approved scene boundary, and the high-fidelity concept must depend on the approved Pencil draft; the concept must bind the frozen visual version/hash, both global reference-effect images, color-token system, restraint/expression profile, functional-UI rules and passed quality rubric before resource/code production. Release the next scene only after all `exit_checks` pass. After every formal loop is complete, schedule one global `integration` task, then one global human `verification` task. Only the global integration task may write through Cocos MCP.
10. Put every Cocos MCP write into one serial `integration` queue. 每个正式场景的首个批次必须先按 `scene_blueprint` 查询/创建缺失节点与组件、设置已声明属性并读回通过，后续资源批次必须按“批量创建/导入资源 → 等待 `refresh-asset` 完成 → 校验 `query-asset-info` / `query-asset-meta`、UUID、类型和子资源 → 脚本批次检查控制台错误 → 应用绑定”执行。资源批次必须声明这些适用能力为前置条件，能力缺失时 `blocked`。Each batch names one `cocos_writer` and requires query, minimal write, refresh evidence when resources change, resource-ready checks, and read-back verification. Do not plan concurrent editor writes, deletion, movement, default overwrite, fixed-delay readiness checks, undocumented compiler-status checks, or use of resources before their batch passes the ready gate.
11. Validate the draft, request human approval, and set `approved` only when approver, time, plan hash, core-gameplay prototype definition, scene-loop coverage, business-flow order, formal-core-gameplay mapping, and capture-manifest hash are complete.
12. Return the plan artifact, capture manifest, frozen input hashes, ownership checks, dependency checks, and unresolved questions through the assigned result path. The orchestrator accepts them before entering `production`.

## Block and invalidate

- If requirements, systems design, technical design, visual direction, project-profile hashes, or any completed scene Pencil/high-fidelity design evidence changes, reject plan reuse and mark the plan and downstream outputs `stale`.
- A decision-changing update to an approved plan without a matching `$grilling` confirmation is `blocked`; never infer that confirmation from an implementation-plan approval.
- Block on missing initial scene, requirements criteria, asset license, concept approval, path ownership, single-writer declaration, scene blueprint, required node, component specification, blueprint read-back assertion, scene-boundary artifact, or scene-boundary human confirmation.
- A plan never replaces a human gate; keep `approval.status` `pending` until explicit approval.

## Output requirements

- `implementation-plan.md` must carry frozen input versions and `sha256:` hashes in YAML front matter and explain the fixed production order, `vertical_slice` prototype gate, `business_flow_levels`, scenes, per-scene `scene_blueprints`, required `scene-boundary-grilling` tasks, prefabs, scripts, approved Pencil-draft and high-fidelity-concept dependencies, asset dependencies, `module_decomposition`, `dependency_graph`, formal `scene_loops` (including `is_core_gameplay`), tasks, ownership, the per-scene asset-approval join gate, and the one global Cocos writer in its body. It must not carry mutable task status, result, or runtime approval fields.
- `capture-manifest.yaml` is a planning artifact, not a verification report. It must bind the frozen profile and visual-direction hashes and enumerate all frozen mobile profiles before baseline capture begins.
- Every production or integration task must have `allowed_paths`, `depends_on`, `acceptance_checks`, and expected outputs.
- Follow the orchestrator result contract: a `passed` result has nonempty evidence, returns only authorized paths, and cannot waive P0 failure.
