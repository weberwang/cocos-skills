# 计划拷问

在实施前，与用户逐项厘清计划或设计中的决策。此 skill 只帮助形成共同理解；在用户明确确认前，不执行该计划，也不改动实现文件。

## Cocos 决策变更门禁

在 `cocos-orchestrate-web-workflow` 中，已批准的 `requirements`、`systems-design`、`technical-design` 与 `planning` 发生会改变目标、范围、规则、架构、约束、依赖或验收标准的变更时，自动进入本 skill。每个正式 `scene_loop` 在资产或代码实现前也必须进入本 skill，专门确认该场景的功能边界；核心玩法原型不适用此附加门禁。用户显式要求 grill 时不受此范围限制。

不要为事实勘误、文案调整、已批准设计的资产生成与集成、代码实现、验证或交付自动进入拷问。先查证这些变更是否确实改变决策；无法查证的事实先探索代码库或工件。

总控派发的决策变更任务会提供 `decision_change.stage` 和 `decision_change.subject_hash`。完成共同理解并获得用户明确确认后，返回与两者一致的 `grilling_confirmation`（阶段、主题哈希、确认者、确认时间和证据路径）。只允许总控写入 `workflow.yaml` 的 `approval_gates.grilling-<stage>`；确认前返回 `blocked`，不产出被修改的阶段工件。

## 先确定拷问对象

1. 查找已提供的计划、设计稿、PRD 及代码库中的相关事实。能从文件、代码、配置、历史记录或测试中确认的事实，直接查找并在提问时说明依据；不要把事实当作问题抛给用户。
2. 若用户从 PRD 开始，直接围绕该 PRD 进行拷问。
3. 若没有 PRD，先拷问 PRD 所需的产品决策，再确认 PRD；不要跳到实现计划或构建。
4. 若已有计划或设计，以该材料为根节点；仅沿着其中实际存在的分支追问，不虚构无关分支。

## 场景功能边界拷问

当任务带有 `scene_boundary` 时，只拷问一个正式 `scene_id`，并读取该场景的 `scene_blueprint`、实施计划、需求、系统与技术设计、冻结视觉方向，以及 `$cocos-plan-project` 的 `scene-boundary-contract.md`。逐项确认：场景目的、进入/退出条件、玩家可执行动作、UI 与交互状态、数据输入/输出与持久化、模块和事件边界、失败/空态/返回路径、明确非范围及可验收标准。场景边界是 Pencil 草图与高保真图的上游输入，不得依赖尚未创建的场景设计稿。

不得用“同其他场景一致”“按常规处理”或未归属的全局功能替代答案；每项功能必须归属到当前场景或明确交给其他稳定 `scene_id`。用户明确确认后，返回 `scene_boundary_confirmation`，其中含 `scene_id`、`scene_loop_id`、`subject_hash`、`confirmed_by`、`confirmed_at`、项目内证据路径和已确认的功能边界。未确认或仍有未决问题时返回 `blocked`，不得开始该场景的 Pencil 草图、高保真图、资源准备或正式代码。

## 追问流程

1. 先建立简短的 design tree：目标与成功标准、范围与非范围、用户流程、约束与风险、依赖与验收。根据材料删去不适用的分支。
2. 从尚未决定且会影响其他决策的最高层分支开始，优先处理阻塞下游的选择。
3. 每轮只问一个决策问题，并附上推荐答案及其简短理由。问题应明确说明它影响哪些后续分支。
4. 等待用户的回答后，记录其决定，更新 design tree，再继续下一个最相关的问题。一次不得提出多个问题或预先列出问题清单。
5. 对用户答案中的事实性缺口，先探索代码库或现有材料；只有无法查证且确实需要用户决定时，才继续提问。
6. 当每个相关分支都已决定或被明确排除时，汇总共同理解：目标、关键决定、范围、约束、依赖、风险与验收标准。
7. 明确询问用户是否确认已达成共同理解。在收到明确确认前，保持在拷问阶段，不执行计划、不开始构建，也不将草案视为批准。

## 提问格式

每轮使用以下紧凑格式，只包含一个问题：

```text
决策：<需要用户决定的事项>
影响：<它阻塞或改变的后续分支>
推荐：<推荐答案>——<简短理由>
问题：<单一、可回答的问题>
```

接受用户选择推荐答案、提出替代方案或要求继续探索。若用户提出替代方案，先确认其与已决定分支的冲突和影响，再进入下一个决策。
